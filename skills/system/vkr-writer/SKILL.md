# VKR Writer — Написание выпускных квалификационных работ

Агентская система для написания ВКР (выпускных квалификационных работ) бакалавриата и магистратуры.
Покрывает полный цикл от выбора темы до подготовки к защите.

---

## Фаза 1: Инициализация проекта

### 1.1 Сбор требований

Собрать у пользователя:
1. **Уровень образования**: бакалавриат (4 главы) или магистратура (6-7 глав)
2. **Направление/специальность**: код специальности по ОКСП
3. **Тема ВКР** или направление исследований
4. **Тема задания**: официальная тема от кафедры (если есть)
5. **Срок сдачи**: дедлайн защиты
6. **Требования кафедры**: особые требования к оформлению (если известны)
7. **Наличие нормоконтроля**: требуется ли проверка структуры

### 1.2 Создание структуры проекта

Создать директорию проекта:

```
vkr-project/
├── meta.json                  # Метаданные проекта
├── plan/
│   ├── timeline.md            # Календарный план
│   └── outline.md             # Развёрнутый план
├── chapters/
│   ├── 00-introduction.md
│   ├── 01-theoretical-chapter.md
│   ├── 02-methodology-chapter.md
│   └── ...
├── literature/
│   ├── bibliography.md        # Библиография (авторизованный список)
│   └── sources/               # Папка с выписками из источников
├── references/
│   ├── gost-2018.md           # Правила оформления по ГОСТ
│   ├── structure-guide.md     # Структура каждой главы
│   └── style-guide.md         # Академический стиль
├── defense/
│   ├── presentation.md        # Текст презентации
│   └── report.md              # Текст доклада
└── output/
    └── draft.docx             # Итоговый документ
```

### 1.3 Инициализация meta.json

```json
{
  "degree": "bachelor" | "master",
  "specialty_code": "XXXX.XX.XX",
  "specialty_name": "...",
  "topic": "...",
  "topic_assignment": "...",
  "deadline": "YYYY-MM-DD",
  "supervisor": "...",
  "created_at": "YYYY-MM-DD",
  "status": "planning",
  "chapters_written": [],
  "plagiarism_score": null
}
```

## Фаза 2: Планирование

### 2.1 Формулировка темы и проблемы

**Цель**: Сформулировать актуальность, проблему, цель и задачи.

**Актуальность** должна содержать:
- Контекст проблемы в современной науке/практике
- Противоречие (теория vs практика, потребность vs наличие решений)
- Последствия нерешения проблемы
- 3-5 источников подтверждений актуальности

**Проблема** формулируется как:
> «Несмотря на [контекст], остаётся нерешённым [проблема], что обусловливает необходимость [решение]»

**Цель** — одно предложение:
> «Целью работы является [разработка/исследование/анализ/совершенствование] ...»

**Задачи** (5-7 штук, соответствуют главам):
1. Провести анализ ... (теория)
2. Рассмотреть методы ... (теория)
3. Разработать/исследовать ... (практическая часть)
4. Обосновать эффективность ... (практическая часть)
5. Разработать рекомендации/внедрить ... (заключение)

### 2.2 Развёрнутый план (outline)

Для каждой главы указать:
- Номер и название
- Подразделы (2-3 уровня вложенности)
- Объём в страницах
- Ключевые источники для каждой секции
- Статус написания

**Стандартная структура бакалавриата:**
```
Введение (1.5-2 стр.)
Глава 1. Теоретический анализ ... (15-20 стр.)
  1.1. Понятие, сущность, история вопроса
  1.2. Классификация, методы, подходы
  1.3. Обзор современных решений/исследований
Глава 2. Исследование/разработка ... (15-20 стр.)
  2.1. Объект и предмет исследования, методология
  2.2. Анализ текущего состояния / разработка решения
  2.3. Результаты исследования / тестирование
Глава 3. Практическая реализация / Экономика (10-15 стр.)
  3.1. Внедрение / описание реализации
  3.2. Оценка эффективности
  3.3. Безопасность / дополнительные аспекты
Заключение (2-3 стр.)
Список литературы (1.5-2 стр.)
Приложения (не ограничено)
```

### 2.3 Календарный план

Разбить дедлайн на этапы:
- Неделя 1-2: Планирование + Введение
- Неделя 3-4: Глава 1 (теория)
- Неделя 5-6: Глава 2 (методология + исследование)
- Неделя 7-8: Глава 3 (практика)
- Неделя 9: Заключение + оформление
- Неделя 10: Нормоконтроль + защита

## Фаза 3: Написание глав

### 3.1 Процесс написания каждой главы

**Правила:**
1. Начинать с теоретических глав, заканчивать практическими
2. Каждая глава пишется отдельно, но с учётом контекста предыдущих
3. Перед написанием — загрузка соответствующего reference файла
4. После написания — проверка структуры и ссылок

**Для каждой главы:**
1. Загрузить `references/structure-guide.md` для понимания структуры
2. Загрузить план главы из `plan/outline.md`
3. Собрать источники из `literature/bibliography.md`
4. Написать текст по подразделам
5. Добавить ссылки на источники в формате [1], [2, с. 15], [3, с. 45-50]
6. Проверить соответствие плану

### 3.2 Введение

Вводится в самом конце, после написания всех глав. Содержит:
- Актуальность темы (3-5 абзацев с ссылками)
- Степень разработанности проблемы (обзор литературы)
- Противоречие, порождающее проблему
- Проблема исследования (1 предложение)
- Объект и предмет исследования
- Цель и задачи (соответствуют плану)
- Методы исследования
- Научная новизна (для магистратуры)
- Теоретическая и практическая значимость
- Позиция, защищаемая в работе
- Апробация результатов
- Структура работы

### 3.3 Теоретическая глава

**Подраздел 1.X — Понятие и сущность:**
- Определения из 3+ источников (сравнительная таблица)
- Эволюция понятия (хронологическая)
- Собственное определение (на основе анализа)

**Подраздел 1.X — Классификация/Методы:**
- Классификационные признаки
- Таблицы классификации
- Сравнительная характеристика подходов

**Подраздел 1.X — Обзор решений:**
- Систематизация современных исследований/решений
- Критический анализ (не просто перечисление!)
- Выявление пробелов

### 3.4 Практическая глава

**Подраздел 2.X — Методология:**
- Обоснование выбора методов
- Описание объекта и предмета
- Методы сбора и анализа данных

**Подраздел 2.X — Исследование/Разработка:**
- Описание методики исследования
- Результаты (таблицы, рисунки, графики)
- Статистическая обработка (если применимо)

**Подраздел 2.X — Результаты:**
- Интерпретация результатов
- Сравнение с теоретическими ожиданиями
- Выявленные закономерности

### 3.5 Заключение

- Краткие выводы по каждой главе (по 2-3 предложения на задачу)
- Подтверждение достижения цели
- Практические рекомендации
- Перспективы дальнейших исследований

## Фаза 4: Оформление

### 4.1 Правила оформления по ГОСТ

Загрузка `references/gost-2018.md` для детальных правил.

**Ключевые требования:**
- Шрифт: Times New Roman, 14 пт, межстрочный интервал 1.5
- Поля: левое 30 мм, правое 15 мм, верхнее 20 мм, нижнее 20 мм
- Абзацный отступ: 1.25 см
- Выравнивание: по ширине
- Нумерация: арабскими цифрами, счётчик с титульника
- Каждый подраздел — с новой страницы
- Рисунки: подпись ПОД рисунком, «Рис. 1 — Название»
- Таблицы: заголовок НАД таблицей, «Таблица 1 — Название»

### 4.2 Оформление ссылок

**Внутритекстовые:**
- Порядковые номера в квадратных скобках: [1]
- При цитировании с указанием страницы: [2, с. 15]
- При цитировании диапазона страниц: [3, с. 45-50]

**Библиографический список (ГОСТ Р 7.0.100-2018):**
```
Монография:
Автор Ф.И. Название книги. — Место издания: Издательство, Год. — 200 с.

Статья:
Автор Ф.И. Название статьи // Название журнала. — Год. — № X. — С. 1-10.

Электронный ресурс:
Автор Ф.И. Название [Электронный ресурс]. — URL: http://... (дата обращения: DD.MM.YYYY).

Нормативный акт:
Название документа. — Дата и номер. — Источник публикации.
```

### 4.3 Список литературы

Правила формирования:
- Не менее 30 источников (бакалавриат), не менее 50 (магистратура)
- Не менее 30% иностранных источников
- Не менее 5 источников за последние 3 года
- Нормативные акты в начале, затем литература, затем интернет-ресурсы
- Алфавитный порядок внутри каждой группы
- Источники на русском и английском — отдельно

## Фаза 4.5: Экспорт в DOCX (критичные требования)

Раздел появился после анализа `vkr-final-v8.docx` (см. `docx-issues-report.md`), где найдено 6 категорий дефектов. Любой генератор `md → docx` обязан выполнять все 12 пунктов ниже.

Подробный чек-лист с python-docx проверками — в `references/docx-export-checklist.md`.

### 4.5.1 Стили абзацев (НЕ «Normal» везде)

**Ошибка-эталон:** все 582 абзаца в v8 имеют стиль `Normal` → поле TOC пустое, нет навигации, заголовки — просто жирный текст.

**Правила:**
1. Создать три пользовательских стиля: `VKR-H1`, `VKR-H2`, `VKR-H3` с атрибутами `outline_level = 0/1/2` (0-based).
2. Для `VKR-H1` обязательно `pageBreakBefore = true` в `pPr` стиля — каждая глава с новой страницы без ручных `<w:br>`.
3. Заголовки, ячейки таблиц, captions, библиография, списки имеют `first_line_indent = 0`.
4. Body paragraphs (обычный текст) получают `first_line_indent = 1,25 см` через стиль `VKR-Body`, а НЕ через модификацию `Normal` (иначе красная строка проникает в ячейки).

### 4.5.2 Структура документа

**Правила порядка:**
1. **Титульник** — без ведущего page break. Первая страница = титульник, не пустой лист.
2. Каждая многострочная подпись титульника — **отдельный абзац** `add_paragraph()`. Никаких `\n` внутри одного run — Word не рендерит это как разрыв строки по центру.
3. Между титульником и оглавлением — один page break.
4. Оглавление вставляется Word-полем `TOC \o "1-3" \h \z \u`, а не вручную. Для наполнения нужны стили Heading с `outline_level`.
5. Перед каждой `ГЛАВА N`, `ЗАКЛЮЧЕНИЕ`, `СПИСОК ИСТОЧНИКОВ`, `ПРИЛОЖЕНИЯ`, `Приложение X` — page break (через `pageBreakBefore` стиля H1).

### 4.5.3 Очистка markdown-артефактов

Парсер `md → docx` обязан **полностью потреблять** markdown-разметку и НЕ пропускать её в финальный текст:

| Что пришло в md | Что должно быть в docx |
|---|---|
| `# Заголовок` | Абзац со стилем `VKR-H1`, текст `Заголовок` (без `#`) |
| `## 1.1 Название` | Абзац `VKR-H2`, текст `1.1 Название` |
| `**жирный**` | Run `bold=True`, текст `жирный` (без `**`) |
| `*курсив*` | Run `italic=True`, текст `курсив` (без `*`) |
| `- пункт` | Абзац списка с маркером `—`, текст `пункт` (без `- `) |
| `1. пункт` | Абзац нумерованного списка, текст `пункт` (без `1. `) |
| `---` | **Ничего** (не эмитить вообще) |
| `![alt](path)` | `add_picture(path)`; если файла нет — плейсхолдер `[рисунок недоступен: path]`, но БЕЗ рекурсивной подписи «Рисунок 1.1 — Рисунок 1.1» |
| `**Таблица N — …**` (перед таблицей) | Ровно **одна** подпись-абзац над таблицей |
| `**Рисунок N — …**` (после картинки) | Ровно **одна** подпись-абзац под рисунком |

**Антипаттерн:** эмитить и «markdown-оригинал», и «свою очищенную подпись» рядом. В v8 это дало 29 дубликатов подписей подряд.

### 4.5.4 Таблицы

**Ошибка-эталон:** 32/32 таблицы в v8 без границ — `<w:tblBorders>` не задан.

**Правила:**
1. Каждой таблице **обязательно** явно задать все 6 границ: `top`, `left`, `bottom`, `right`, `insideH`, `insideV` (single, sz=4, color=000000).
2. Порядок обязателен: сначала добавить `<w:tblPr>` если его нет, потом вставить `<w:tblBorders>`; иначе свойство проигнорируется.
3. Ширина таблицы — `<w:tblW w:w="5000" w:type="pct"/>` (100% от полосы набора A4), не `auto` — иначе таблицы «плавают» и выходят за поля.
4. Ячейки шапки — с заливкой `F2F2F2` и `bold=True`.
5. Все абзацы в ячейках имеют `first_line_indent = 0`, `line_spacing = 1.15`, `space_before/after = 0`. Это обязательно, даже если в body-стиле стоит красная строка.
6. Числовые ячейки — выравнивание `RIGHT`, текстовые — `LEFT`. Заголовки — `CENTER`.
7. Текст ячеек пропустить через `_append_inline()` (обработать `**` / `*`) — а не вставлять «как есть».

### 4.5.5 Рисунки

1. `![alt](path)` → `add_picture` с `width = Cm(16)` (ширина полосы набора A4).
2. Если файл не найден — плейсхолдер `[Рисунок недоступен: <path>]` курсивом, 12 pt, серый. Подпись под ним — ОДИН раз.
3. Подпись формата `Рисунок N.M — Название` (тире U+2014, не дефис), по центру, без жирного, 14 pt. Счётчик сквозной внутри главы.

### 4.5.6 Библиография

**Ошибка-эталон:** в v8 записи — плоский нумерованный список `Normal` без висячего отступа.

**Правила:**
1. Каждая запись — отдельный абзац со стилем `VKR-Biblio`.
2. `left_indent = 0.75 см`, `first_line_indent = -0.75 см` (висячая строка) — так номер «вылезает» влево, текст записи выровнен по левому краю после номера.
3. `first_line_indent` красной строки body не применять.
4. Без маркированного списка Word — только ручная нумерация `1. `, `2. ` и т.д. в начале текста (ГОСТ требует арабскую нумерацию).

### 4.5.7 Приложения

**Ошибка-эталон:** в v8 приложения А/Б/В/Г идут подряд без разрывов, `---` оставлен как разделитель, каждая таблица имеет **двойной номер** (`Таблица А.1` и `Таблица 3.13`).

**Правила:**
1. Каждое приложение начинается с нового page break.
2. Первая строка — «ПРИЛОЖЕНИЕ А» заглавными, по правому краю.
3. Вторая строка — название приложения, по центру, жирным.
4. Нумерация таблиц/рисунков **внутри приложений** — `А.1`, `А.2`, `Б.1` и т.д. Сквозные номера `3.13`, `3.14` в приложения НЕ переходят — это смысловая ошибка.
5. Никаких `---` как разделителей между приложениями.

### 4.5.8 Поля страницы и шрифт

- Шрифт: Times New Roman, 14 pt (таблицы 12 pt допустимо).
- Межстрочный интервал body: 1.5, таблицы: 1.15.
- Поля: L=3, R=1, T=2, B=2 см.
- Нумерация страниц в правом верхнем углу, со 2-й страницы (на титульнике — нет). Используется `<w:titlePg/>` в sectPr.

### 4.5.9 Язык документа

**Ошибка-эталон:** в первой версии v9 python-docx создал документ без `<w:lang>` — Word трактовал кириллицу как «неопределённый язык», подчёркивал весь текст красным, переносы слов ломались.

**Правила:**
1. В `styles.xml` прописать `<w:docDefaults>/<w:rPrDefault>/<w:rPr>/<w:lang w:val="ru-RU" w:eastAsia="ru-RU" w:bidi="ar-SA"/>`.
2. Продублировать `<w:lang>` в ключевых стилях: `Normal`, `VKR-H1/H2/H3`, `VKR-Body`, `VKR-Cell`, `VKR-Biblio`, `VKR-Caption` — чтобы переопределение Normal не сломало русский.
3. Для англоязычных цитат (фрагменты вроде `*to control*`, названия иностранных источников) явно ставить `<w:lang w:val="en-US"/>` на нужном run — иначе Word пометит их как «ошибка орфографии».

**XML-сниппет (python-docx):**

```python
from docx.oxml import OxmlElement
from docx.oxml.ns import qn

def set_document_language(doc, lang="ru-RU"):
    styles = doc.styles.element
    dd = styles.find(qn("w:docDefaults")) or _make_child(styles, "w:docDefaults", 0)
    rpd = dd.find(qn("w:rPrDefault")) or _make_child(dd, "w:rPrDefault", 0)
    rpr = rpd.find(qn("w:rPr")) or _make_child(rpd, "w:rPr")
    for old in rpr.findall(qn("w:lang")):
        rpr.remove(old)
    el = OxmlElement("w:lang")
    el.set(qn("w:val"), lang)
    el.set(qn("w:eastAsia"), lang)
    el.set(qn("w:bidi"), "ar-SA")
    rpr.append(el)
```

## Фаза 5: Поиск источников

### 5.1 Стратегия поиска

**Приоритетные источники:**

## Фаза 8: Пост-валидация DOCX

После каждой сборки `md → docx` прогнать `scripts/docx_linter.py` — он проверяет все 6 категорий дефектов из Фазы 4.5 и возвращает JSON-отчёт.

### Запуск

```bash

```

Выход:
- exit code `0` — все критичные проверки пройдены;
- exit code `1` — найдены критичные дефекты (пустая первая страница, markdown-артефакты, все `Normal`, невидимые таблицы, двойные подписи, первая страница не с номера 1);
- exit code `2` — только warnings (отсутствие красной строки, отсутствие висячих отступов в библио и т.п.).

### Что проверяет (6 категорий)

| # | Категория | Критичность |
|---|-----------|-------------|
| 1 | Первая страница (нет ведущего page break, есть титульник) | critical |
| 2 | Markdown-артефакты в тексте (`^#\s`, `\*\*[^*]+\*\*`, `^---$`, `\\n` в run) | critical |
| 3 | Стили (нет ≥ 90% `Normal`, есть `Heading 1/2/3` или аналоги) | critical |
| 4 | Таблицы: `tblBorders` на каждой, ширина задана, `first_line_indent=0` в ячейках | critical |
| 5 | Дубликаты подписей (подряд идущие идентичные абзацы `Таблица…` / `Рисунок…`) | critical |
| 6 | Красная строка body, висячий отступ библио, page breaks перед главами | warning |

## Антипаттерны DOCX-генератора

Конкретные ошибки, найденные в `vkr-final-v8.docx`. При написании любого `md → docx` конвертера избегать каждую из них:

1. **Пустой первый абзац с `<w:br w:type="page"/>`** перед титульником. Убрать ведущий break — первая страница и так первая.
2. **Многострочные абзацы через `\n`:** `add_paragraph("АНО\nВО «ЛАДИРО»")` — Word рендерит это как single paragraph с soft-break, выравнивание ломается. Использовать отдельные `add_paragraph()` на каждую строку.
3. **Стиль `Normal` для всего подряд.** Без `outline_level` не работает поле TOC и навигация. Создать `VKR-H1/H2/H3`.
4. **Забытый `<w:tblBorders>`.** Проверка: открыть `document.xml` и поискать `w:tblBorders` — он должен встречаться ровно `N_tables` раз.
5. **Дублирующиеся подписи таблиц/рисунков** — эмитить и markdown-оригинал, и «очищенную копию». Потребляйте токен `**Таблица X — …**` один раз.
6. **Placeholder `[изображение недоступно]` + подпись «Рисунок X — Рисунок X»** (рекурсия). Подпись — один раз, на основе markdown `**Рисунок X — …**`.
7. **`---` в теле документа** как горизонтальная черта или разделитель приложений. В docx `---` не означает ничего — пропускать при парсинге.
8. **Hard-coded `[1]`, `[2, с. 15]` без проверки, что номер существует в bibliography.** Чекер должен падать, если ссылка указывает на несуществующий номер.
9. **Библиография без висячего отступа.** Номер «1.» вылезает влево от текста, многострочные записи выравнены по левому краю после номера.
10. **Приложения со сквозной нумерацией таблиц главы 3** (`Таблица 3.13` в Приложении А). Должно быть `Таблица А.1`.
11. **Ширина таблицы `auto`.** Таблица расплывается или сжимается, не вписывается в поля. Использовать `pct=5000`.
12. **Таблицы содержимого собраны руками** (абзацы с текстом `ВВЕДЕНИЕ`, `ГЛАВА 1…` без табов, лидеров и номеров страниц). Только поле `TOC`.
13. **Документ без установленного языка** — Word считает кириллицу «неопределённым языком», красные подчёркивания под всем текстом, переносы не работают. Требуется `<w:lang w:val="ru-RU"/>` в `docDefaults` + ключевых стилях (см. 4.5.9).

Каждый раз при регенерации DOCX прогонять `docx_linter.py` и требовать exit code 0.

Рентабельность выросла с 8.0% до 13.3% (+3.3 п.п.).

Гипотеза 1: Оптимизация производственных затрат (снижение себестоимости).

- Не загружать все reference одновременно
- Использовать context_keeper.py для валидации согласованности

- Перед генерацией каждой главы — запускать `context_keeper.py validate`
- После генерации каждой главы — запускать `derobotizator.py` и `logic_inspector.py`
