# tools / Опциональные скрипты

Этот каталог хранит **воспроизводимые вспомогательные скрипты**. Основной процесс навыка — `SKILL.md` и `prompts/`; этот каталог фокусируется на выполнимых инструментах, таких как конвертация форматов.

## Проверка новизны в ФИПС (Роспатент) — fips.ru (Шаг 5 приоритет)

| Скрипт | Назначение |
|--------|-----------|
| **`fips_epub_search.py`** | **Шаг 5 приоритет**: один шаг — загрузка + парсинг, **без записи HTML на диск**; **Agent должен по `prior_art_search.md` вызывать последовательно, по одному блоку и самостоятельно объединять JSON**; скрипт также объединяет при **множественных терминах в одной команде** (ручная/локальная удобство); **stdout — одна строка** `EPUB_HITS_JSON:`; stderr `EPUB_*` — **ASCII**; UTF-8 / PowerShell — см. **INSTALL.md**. |
| **`fips_epub_crawler.py`** | Только Playwright — загрузка и **по умолчанию сохранение** HTML результата; stdout также содержит **`EPUB_HITS_JSON:`**. |
| **`fips_epub_parse.py`** | Только парсинг сохранённого HTML: `python tools/fips_epub_parse.py path/to/_last_result_xxx.html`; поля: название, номер публикации, ссылка, **`abstract`** (если есть). |

Зависимости: `pip install -r tools/requirements-fips.txt` и `python -m playwright install chromium`. Переменные окружения — в шапках скриптов. По умолчанию HTML результатов падает в **`tools/_last_result_*.html`** (уже в `.gitignore`).

При ошибке парсинга или отсутствии результатов — Agent по **`prompts/prior_art_search.md`** переходит к **WebSearch** (Google Scholar / Google Patents).

## mermaid_render.py — mermaid: диаграмма → PNG + финальный Markdown + **по умолчанию Word**

Каждый fenced **mermaid** (`` ```mermaid`` ``) передаётся **`mmdc`** для рендеринга в PNG; в выходном `.md` — ссылки типа ``mermaid_figures/fig_001.png``, которые **`md_to_docx.py`** встраивает в Word. **3.2 Блок-схема системы** и **3.4 Блок-схема процесса** — через mermaid (`flowchart` / `subgraph` и т.д.); тело заявки **не** требует отдельных ASCII-диаграмм или PlantUML.

**Понижение при ошибке генерации**: если `mmdc` для одного блока **не succeeds** — **не останавливается**; на этом месте **сохраняется** оригинальный `` ```mermaid`` … `` ``` ``; остальные блоки — по-прежнему. Пишет финальный `.md`, и **по-прежнему пытается** создать Word (не-рендеренные блоки в Word — **Consolas код-блок**, согласовано с `md_to_docx`).

### Зависимости: mermaid (Node.js + `mmdc`)

| Способ | Установка | Примечание |
|--------|----------|-----------|
| **Локальный npm (рекомендуется)** | **Node.js** + `npm install` в этом каталоге (см. `package.json`) | Приоритет `tools/node_modules/.bin/mmdc`, избежать `npx` каждый раз |
| **npx** | Когда `npm install` не выполнен, скрипт вызывает `npx -y @mermaid-js/mermaid-cli mmdc` | Первый раз может быть медленно |
| **Глобальный npm** | `npm install -g @mermaid-js/mermaid-cli` | Даёт `mmdc` в **PATH** |

mermaid скрипт ищет: `tools/node_modules/.bin/mmdc` → **PATH** `mmdc` → `npx`.

Для создания Word: `pip install -r requirements.txt` (независимо от таблицы выше).

**Рекомендуемый npm (локальный CLI)**:

```bash
cd tools
npm install
```

`package.json` уже содержит **`puppeteer`** (peer для `@mermaid-js/mermaid-cli`). **Puppeteer 23+** может не загружать браузер при `npm install`; если проверка или `mmdc` сообщают **Could not find Chrome**, в **`tools/`** выполнить:

```bash
npx puppeteer browsers install chrome-headless-shell
```

(или по ошибке выбрать `chrome`; см. [Puppeteer документацию](https://pptr.dev/).)

### mermaid CLI и ручная проверка

**`mermaid_render.py`** — **совместим с 11.x**: в **`mmdc -i <.mmd> -o <.png> -b white`** по умолчанию добавляется **`-s 2 -w 1400 -H 1050`** (выше разрешение и viewport, блок-схема в Word чётче). Для дополнительной резкости — **`--mmdc-scale 3`** (PNG больше); для возврата к старому — **`--mmdc-scale 1 --mmdc-width 800 --mmdc-height 600`**.  
Если где-то написано `npx -y @mermaid-js/mermaid-cli -i …`, **пропущен подкоманду `mmdc`**, параметры сдвинутся; правильный пример:

```bash
npx -y @mermaid-js/mermaid-cli mmdc -i sample.mmd -o sample.png -b white
```

Можно создать `sample.mmd` (например `flowchart LR; A-->B`) для проверки; если PNG выходит — **mmdc + Chrome** работает, иначе установить **`puppeteer` браузер**.

### Использование

```bash
# Написать финальный .md и в том же каталоге同名 .docx (по умолчанию); -o должно быть «название_изобретения_YYYYMMDDHHmmss.md» (см. prompts/disclosure_builder.md §7.3 пункт 5)
python3 tools/mermaid_render.py -i draft.md -o "название_изобретения_20260512103000.md"

# Указать путь .docx (.md имя должно содержать timestamp)
python3 tools/mermaid_render.py -i draft.md -o out/название_изобретения_20260512103000.md --docx out/название_изобретения_20260512103000.docx

# Только Markdown, без Word
python3 tools/mermaid_render.py -i draft.md -o "название_изобретения_20260512103000.md" --no-docx

# Более высокая чёткость (опционально)
python3 tools/mermaid_render.py -i draft.md -o "…финальный.md" --mmdc-scale 3 --mmdc-width 1600 --mmdc-height 1200

# Указать подкаталог mermaid-изображений (относительно выходного .md)
python3 tools/mermaid_render.py -i draft.md -o out/название_изобретения_20260512103000.md --assets-dir figures/mermaid
```

**Ошибка создания Word** (отсутствуют зависимости, ошибка макета и т.д.): скрипт завершается с кодом **0** (Markdown уже успешен); stderr выводит **команду для ручного запуска `md_to_docx.py`** — скопировать и выполнить.

На Windows, если установлен только Node без `npm install`, скрипт вызывает `npx -y @mermaid-js/mermaid-cli mmdc` (первый раз может быть медленно).

### Связь с заявкой

- Навык требует финальной доставки **Markdown + Word одновременно**, и **`-o` имя должно содержать `_{YYYYMMDDHHmmss}`** (`prompts/disclosure_builder.md` §7.3 пункт 5, включая первичную доставку); **3.2 Блок-схема** и **3.4 Блок-схема процесса** — fenced mermaid, **не** ASCII-диаграммы.
- Перед доставкой поверенному: выполнить `mermaid_render.py` один раз (по умолчанию вызывает `md_to_docx.py`); если Word ошибка — по stderr выполнить `md_to_docx.py` вручную.

## md_to_docx.py — Markdown → Word

Конвертирует Markdown заявки в `.docx`, **`#`–`######` → Word заголовки 1–9**, тело — шрифт 宋体 10.5pt, код-блоки — Consolas, для поверенного или для редактирования в Word.

**Диаграммы**: финальная версия использует **`mermaid_render.py`** для конвертации mermaid в PNG; если некоторые блоки не сработали и围栏 сохранены, этот скрипт запишет **сохранённые围栏** в Word как **код-блоки**. Скрипт **не** вызывает `mmdc`.

### Зависимости

```bash
pip install -r requirements.txt
```

Зависимость — `python-docx` (см. корневой `requirements.txt`).

### Использование

```bash
python3 tools/md_to_docx.py --input path/to/заявка.md --output path/to/заявка.docx
```

Изображения `![](относительный_путь.png)`: по умолчанию относительно **каталога Markdown-файла**; можно указать корень:

```bash
python3 tools/md_to_docx.py -i ./outputs/case/заявка.md -o ./outputs/case/заявка.docx --base-dir ./outputs/case
```

**Рисунки**: для PNG/GIF/JPEG — считывает пиксельные размеры, масштабирует **сохраняя пропорции** в **макс. ширина 5.5" × макс. высота 8.2"**, одновременно задаёт `width`/`height`, чтобы вертикальные блок-схемы не растягивались по ширине и не выходили за границы страницы. Можно настроить по полям, например:

```bash
python3 tools/md_to_docx.py -i a.md -o a.docx --image-max-width-inches 6 --image-max-height-inches 9
```

В Claude Code можно заменить `tools` на `${CLAUDE_SKILL_DIR}/tools`.

### Поддерживаемый подмножество Markdown

| Элемент | Поведение |
|---------|----------|
| `#`–`######` | Заголовки Word 1–9 |
| Параграф | Текст основного шрифта, `**жирный**`, `` `inline code` ``; **соседние непустые строки (без пустой между) — отдельные параграфы**, «(1)…(2)…» — на отдельных строках |
| `-` / `*` список | Маркированный список |
| `1.` список | Нумерованный список |
| ` ``` `围栏 | Код-блок |
| `\| таблица \|` | Простая таблица (Table Grid) |
| `> ` | Цитата с отступом |
| `---` и т.д. | Светлый разделитель |
| `![](путь)` | Встроенное изображение (путь должен существовать; пропорции внутри макс. ширины/высоты) |

**Не полностью поддерживается**: сложные вложенные списки, HTML-блоки, **нерендеренные围栏** mermaid (код-блок), сноски, чеклисты и т.д. Перед финализацией запустить **`mermaid_render.py`**; если только внешние PNG — писать `![](...)` напрямую.

### Пояснения к макету (md_to_docx)

- В разных версиях Word «Заголовок 1» может отображаться как «Heading 1» или «Заголовок 1»; стиль — заголовок уровня大纲, доступен в навигации и оглавлении.
- Если нужен фиксированный шаблон (колонтитулы, первая страница), можно применить `.dotx` после генерации или расширить `python-docx` для открытия шаблона.

## iteration_dialog_log.py — Лог итераций (для итерации)

После каждой доставки **`merger.md` / `correction_handler.md`** — в **каталоге проекта** добавить запись **`Лог_итераций_заявки.md`**: **локальное время и UTC**, сводка пояснения, имя доставки, выдержка сводки. Правила — в **`prompts/iteration_context.md`**.

**Зависимости**: только стандартная библиотека.

```bash
python3 tools/iteration_dialog_log.py --case-dir outputs/Проект --kind merge \
  --user "Добавил материалы по планировщику задач, объединил в раздел III" \
  --summary "Расширил 3.4, обновил примеры; не изменил объекты охраны." \
  --artifacts "название_изобретения_20260512103000.md,название_изобретения_20260512103000.docx"
```

- `--kind`: `merge` или `correct`.  
- `--log-name`: опционально, по умолчанию `Лог_итераций_заявки.md`; для англоязычной среды — `disclosure_revision_log.md`.  
- При невозможности выполнить скрипт — Agent вручную по той же структуре.

## docx_to_md.py — Word → Markdown + извлечение изображений

Конвертирует **.docx** (Word / WPS и т.д.) в **Markdown**, сохраняет встроенные изображения на диск, для **`Read` и сканирования Шага 2** (гораздо надёжнее, чем чтение бинарного .docx). **Шаг 2** для **каждого** `.docx` в дереве сканирования должен сначала конвертировать, затем читать `.md` — см. `prompts/project_scan.md`.

### Зависимости

Совместно с `md_to_docx` — корневой `requirements.txt` (`python-docx` + **`mammoth`**).

```bash
pip install -r requirements.txt
```

### Использование

```bash
python3 tools/docx_to_md.py --input path/to/описание_проекта.docx --output outputs/case/описание.md
```

- По умолчанию каталог изображений: `outputs/case/описание_media/`, ссылки — `![](описание_media/img_0001.png)`.
- Пользовательский каталог изображений:

```bash
python3 tools/docx_to_md.py -i ./raw/spec.docx -o ./knowledge/spec.md --media-dir ./knowledge/spec_assets
```

Предупреждения конвертации (стили, WMF-изображения и т.д.) — в **stderr**, но `.md` всё равно генерируется.

### Ограничения (mammoth)

- Только **`.docx`** (OOXML); старые **`.doc`** не поддерживаются.
- **Markdown-вывод помечен как deprecated** в mammoth; сложный макет может быть хуже, чем «сначала HTML, потом MD»; для патентного сканирования обычно достаточно. Если макет сломан, рекомендуется экспорт в PDF или текст.
- **WMF/EMF** Windows-метафайлы — возможно, требуют отдельной обработки (см. [mammoth WMF guide](https://github.com/mwilliamson/python-mammoth)).

В Claude Code можно заменить `tools` на `${CLAUDE_SKILL_DIR}/tools`. На Windows без `python3` — использовать `python`.

## pptx_to_md.py — PowerPoint → Markdown + извлечение изображений

Конвертирует **.pptx** / **.ppsx** по **слайдам** в Markdown, извлекает **встроенные растровые изображения** (PICTURE shape), для **`Read` и сканирования Шага 2**. **Шаг 2** для **каждого** `.pptx` в дереве сканирования должен сначала конвертировать, затем читать `.md` — см. `prompts/project_scan.md`.

### Зависимости

**`python-pptx`** из корневого `requirements.txt`.

```bash
pip install -r requirements.txt
```

### Использование

```bash
python3 tools/pptx_to_md.py --input path/to/материалы_для_проверки.pptx --output outputs/case/проверка.md
```

- По умолчанию каталог изображений: `outputs/case/проверка_media/`, имена — `slide03_img0001.png`.
- Пользовательский каталог изображений:

```bash
python3 tools/pptx_to_md.py -i ./raw/deck.pptx -o ./knowledge/deck.md --media-dir ./knowledge/deck_media
```

Каждая страница — заголовок `## Слайд N`, затем **текст и таблицы** (упрощённые pipe-таблицы) и ссылки на изображения; **заметки докладчика** — раздел «Заметки» в конце страницы.

### Ограничения (python-pptx)

- Только **`.pptx` / `.ppsx`** (OOXML); **`.ppt`** не поддерживается, сначала сохранить.
- **Диаграммы, SmartArt, embedded OLE** — если не сохранены как обычные изображения, **не** растрятся в PNG; можно сначала сохранить как изображение или PDF.
- Текст выводится в порядке обхода shapes, может немного отличаться от визуального порядка чтения.

В Claude Code можно заменить `tools` на `${CLAUDE_SKILL_DIR}/tools`. На Windows без `python3` — использовать `python`.
