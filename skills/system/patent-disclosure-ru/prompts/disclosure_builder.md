# Шаблон технической заявки (Шаг 7)

Подробные примеры разделов и **mermaid**-диаграммы — в **`template_reference.md`** этого каталога.

## 7.1 Структура разделов

```
1. Примечания
2. I. Описание предшествующего уровня техники и выявленных недостатков
    - 1.1 Предшествующий уровень техники (по техническим направлениям, с результатами поиска в ФИПС)
    - 1.2 Недостатки предшествующего уровня техники
3. II. Техническая задача, решаемая изобретением
4. III. Подробное описание изобретения
    - 3.1 Предпосылки
    - 3.2 Блок-схема системы (**mermaid**, `flowchart` + `subgraph` для слоёв; перед доставкой — PNG; **не** ASCII-схемы)
    - 3.3 Описание функций модулей (роль и взаимосвязи, не вход/выход)
    - 3.4 Описание процесса системы (**mermaid-блок-схема**, перед доставкой — PNG; **не** ASCII-схемы)
    - 3.4.1 Подразделы (если есть ключевые инновации)
    - 3.5 Ключевые параметры
5. IV. Преимущества изобретения по сравнению с предшествующим уровнем техники
6. V. Ключевые технические моменты и объекты охраны
7. VI. Прочее (примеры реализации, технические эффекты, примеры параметров)
```

**Жёсткие требования к 1.1**: в **каждой** публикации (патент или статья) в конце или в таблице — **подтверждённая ссылка на источник** (формат — в `prior_art_search.md` и `template_reference.md` §1.1). При наличии **`abstract`** в результатах ФИПС: описание **«техническая схема / применение / ограничения»** должно быть **на основе понимания abstract** (переработано, не дословно), **не** противоречить абстракту; см. **`prior_art_search.md`** «`abstract` обязателен».

**Запрещено**: включать чеклист самопроверки в тело заявки; самопроверка — внутренняя, не в доставке.

**Запрещено**: в конце Markdown — ссылки на репозиторий навыка, `examples/`, «учебный пример», «вымышленный», «не является юридическим обязательством» и т.п. Тело заявки **завершается на разделе VI** — **не** добавлять сноски в конец.

## 7.2 Шаблон шапки документа

```markdown
# Техническая заявка на изобретение

**Название изобретения**: [по запросу] Способ и система XXX

**Контактное лицо**:
- ФИО: [по запросу]
- Телефон: [по запросу]
- Email: [по запросу]

**Тип защиты**: Изобретение

---

## Примечания

(1) Заявка должна быть понятна патентному поверенному, особенно описание предшествующего уровня техники и подробное описание изобретения — необходимо писать полно, ясно, полностью;
(2) Степень раскрытия техники должна соответствовать уровню специалиста в данной области, способного осуществить изобретение без творческого труда;
(3) При обращении к поверенному по техническим вопросам — предоставлять ответы и пояснения, своевременно и корректно дополнять технические материалы.
```

## 7.3 Именование выходных файлов (каталог пользователя, обязательно)

При записи технической заявки в каталог пользователя (например `outputs/{идентификатор_проекта}/`):

1. **Извлечь**: из строки `**Название изобретения**:` взять полное название; убрать плейсхолдеры (`[по запросу]`, `【по запросу]`), пробелы по краям.
2. **Нормализовать**: удалить/заменить недопустимые символы Windows `\ / : * ? " < > |` и переносы; сжать множественные пробелы.
3. **Длина**: имя (без расширения) — **≤ 80 символов**; при превышении — **обрезать** до ~80, сохраняя семантику (например до «способ и система»), **не** использовать случайную строку.
4. **Команда финальной доставки**: `mermaid_render.py` `-o` / `--docx` — **то же имя**; см. пункт 5 ( **название + `_` + timestamp 14 цифр**).
5. **Timestamp (обязательно для всех доставок)**: любой `.md`/`.docx`, доставленный пользователю — **`{нормализованное_название}_{YYYYMMDDHHmmss}.md`** и同名 **`.docx`**:
   - **Первичная** и **итерация** — **одно правило**; timestamp — **локальное время** в формате ГГГГММДДЧЧммсс.
   - **Не перезаписывать** существующие файлы; каждый раз — **новый timestamp**, новое имя. Пользователь просит перезаписать — следовать.
   - `mermaid_render.py` `-o` и `--docx` — **то же имя**.
   - **Исключения**: (1) `examples/` — может использовать фиксированное имя; (2) пользователь указал имя — следовать (но timestamp рекомендуется).

Agent при записи использует это правило и указывает путь в ответе.

## 7.4 Блок-схема и блок-схема процесса

- **Только fenced mermaid** (локальный `mmdc`, не требует PlantUML); Word — PNG; **не** ASCII-схемы
- Названия узлов — короткие, русские/числа, без ①②③. Допустимо краткое текстовое описание шагов; **не** заменять ASCII-схемой

## 7.5 Требования обезличивания

| Тип | Способ обезличивания | Пример |
|-----|---------------------|--------|
| Бизнес/отрасль | Обобщить | «XX-контроль» → «многоклассовая классификация» |
| Конкретные классы | Заменить на A/B/C | «Класс1, Класс2» → «КлассA, КлассB, КлассC» |
| Конкретные числа | Указать диапазон или обобщить | «N человек ежедневно» → «определённый объём ежедневно» |
| Компания/продукт | Не указывать конкретные названия | Удалить или заменить на «некая система» |

## Финальная доставка: Markdown + Word (обязательно)

**Версия и перезапись**: все доставки — по правилу **§7.3 пункт 5** «**название + timestamp**». В одном каталоге — история версий. **Опционально**: `versions/` для ручных снимков.

**Обязательно** доставить одновременно:

1. **Markdown**: все mermaid — как PNG-ссылки `![](…png)` (от `mermaid_render.py`), **не** содержать围栏 `` ```mermaid``.
2. **Word**: запустить `md_to_docx.py` на предыдущий файл, или одной командой:

   ```bash
   python3 tools/mermaid_render.py -i "черновик_с_схемами.md" -o "название_изобретения_20260512103000.md"
   ```

   По умолчанию —同名 `.docx` в том же каталоге; `--docx` для указания пути; `--no-docx` для пропуска.

(Зависимости: Node/mmdc, `pip install -r requirements.txt` — см. `tools/README.md`.)
