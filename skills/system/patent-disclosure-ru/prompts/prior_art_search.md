# Поиск предшествующего уровня техники (Шаг 5)

## Когда обязательно

**Обязательно** до генерации полной технической заявки или **в процессе**; результаты записи в раздел **1.1 Предшествующий уровень техники** и **сравнительный анализ** с данным решением.

## Источники поиска (**приоритетно ФИПС (Роспатент), затем WebSearch**)

### A. ФИПС (Роспатент) — **приоритет**, официальный источник

1. **Ресурс**: [ФИПС — Федеральный институт промышленной собственности](https://fips.ru/) (официальный сайт Роспатента).
2. **Инструменты**:
   - **`fips_epub_search.py`** — одношаговый поиск и парсинг результатов (Playwright); HTML **только в памяти**, **не** на диск.
   - **`fips_epub_crawler.py`** — для полных URL-запросов.
   - **`fips_epub_parse.py`** — парсинг результатов.
3. **Поисковые запросы (обязательно до генерации команды)**

   **Обязанность Agent**: в **момент генерации команды** собрать **2–8 релевантных поисковых единиц** из технической схемы, патентных решений или темы пользователя, **только ASCII пробелы**, затем записать в параметры `fips_epub_search.py`. Каждая единица — **семантический блок с поисковой ценностью**: **профессиональный термин**, **именная конструкция**, **существительное+глагол** (например «многоагентная система», «квантовое вычисление»), **устоявшееся сочетание**; **не** дробить на отдельные иероглифы, **не** склеивать в одно длинное предложение.
   - **Запрещено** передавать **целую длинную фразу без пробелов** как единственный параметр (например `".../fips_epub_search.py" "поиск патентных решений в проектной документации"`).
   - **Agent**: **каждый вызов — один поисковой блок** (одна фраза за раз); **2–8 единиц = 2–8 отдельных вызовов**, **не** передавать несколько единиц за один вызов (скрипт поддерживает, но только для локального/ручного использования).
   - Пример (три вызова, по одному блоку):

     ```bash
     python3 …/fips_epub_search.py многоагентная система
     python3 …/fips_epub_search.py квантовое вычисление
     python3 …/fips_epub_search.py распределённая обработка
     ```

   - Скрипт **не** делает авто-токенизацию; для полного предложения AND — использовать **`fips_epub_crawler.py`**.

4. **Запуск (после чтения этого файла — сначала попробовать)**:

   ```bash
   pip install -r tools/requirements-fips.txt
   python -m playwright install chromium
   # Agent: для каждого поискового блока — отдельный вызов (пример: первый)
   python3 ${CLAUDE_SKILL_DIR}/tools/fips_epub_search.py блок_1
   ```

   - **Обязанность Agent — объединить**: разобрать **stdout** — единственная строка **`EPUB_HITS_JSON:`** + JSON-массив; объединить по **`pub_number`** (первичный ключ) или **`link`** (вторичный) или **название** (резервный).
   - При ручном множестве блоков — скрипт по умолчанию объединяет; **stderr** может содержать **`EPUB_MERGE:`**.
   - Успех: **stdout** — одна строка **`EPUB_HITS_JSON:`** + JSON-массив (UTF-8, включая русский `abstract`); **`EPUB_NOTE:`** / **`EPUB_HINT:`** в **stderr** (ASCII); **не** принимать stderr или кодировку терминала за «нет результатов».
   - Записать в чек-лист и 1.1: номер публикации, название, **ссылка на ФИПС** (см. ниже — `abstract` обязателен).
   - **Условия перехода к B**: не 0-код, таймаут, нет Playwright, **`EPUB_HITS_JSON` — пустой массив**, или результаты явно не релевантны.

5. **Поле `abstract` (обязательное для российских публикаций)**

   Если у записи в **`EPUB_HITS_JSON`** есть непустой **`abstract`** (извлечён из страницы ФИПС), для **этой публикации** строго следовать:

   - **Обязательно**: чек-лист и 1.1 должны включать **анализ технической схемы, приложения и ограничений** — **только на основе полного понимания `abstract`**; **запрещено** изобретать на основе заголовка, номера публикации или URL.
   - **Полное понимание**: перед записью в 1.1 или чек-лист — Agent должен **в процессе рассуждения** определить: **техническая область**, **какую проблему решает**, **ключевые элементы/модули**, **основной эффект или процесс**; если заголовок и abstract расходятся — **abstract приоритетнее**.
   - **Представление в тексте**: **не** копировать абстракт дословно (избежать плагиата и превышения объёма); **переработать** в «описание схемы + применение + недостатки/ограничения»; чек-лист может содержать более длинную выдержку, но с пометкой «источник — абстракт ФИПС».
   - **Отсутствие**: если у записи **нет** `abstract` или пустой (старая страница / таблица не извлечена) — пометить в чек-листе «нет поля abstract», использовать **детальную страницу** или **Google Patents** для подтверждения, **не** оставлять без объяснения.

6. **Ссылки и библиографическое описание**: ссылки ФИПС — только от скрипта; **запрещено** изобретать; если доступен только номер публикации от ФИПС — добавить **Google Patents** `https://patents.google.com/patent/RU…/en` как **дополнительный** источник (всё равно проверить открытием).

### B. Google Patents и Google Scholar — **снижение / дополнение**

Активировать при **недоступности A или недостатке результатов**:

1. **Российские патенты (Google Patents)**: [Google Patents](https://patents.google.com/) — поиск по названию, заявителю или номеру публикации (RU…, RU…C1, RU…A и т.д.). Использовать **стабильные библиографические URL**.
2. **Научные статьи**: [Google Scholar](https://scholar.google.com) — ключевые термины, названия методов, области применения; комбинация 2–3 поисковых запросов.
3. **Другие источники**: английские статьи, международные патенты (WIPO/PCT, EPO) — Google Patents, DOI, arXiv, официальные сайты издательств.
4. **Конструкция запросов**: ключевые термины технического решения, области применения и названия методов.

### C. Другие российские источники (дополнительно)

- **WIPO PATENTSCOPE**: [https://patentscope.wipo.int](https://patentscope.wipo.int) — международные патентные заявки (PCT), включая российские заявки PCT.
- **EPO Register**: [https://register.epo.org](https://register.epo.org) — европейские патенты, полезные для сравнения.
- **Базы научных статей**: КиберЛенинка, eLibrary.ru — для научных статей по теме.

## Требования к анализу

Для каждой релевантной публикации или статьи:

- Номер публикации / идентификатор документа
- Ключевые технические моменты (если ФИПС `abstract` — **на основе понимания abstract**, см. выше)
- Область применения
- **Ограничения**
- **Ссылка на источник (обязательно)**: каждая публикация должна иметь **как минимум одну публичную ссылку**, согласованную с библиографическими данными; **запрещено** изобретать URL; перед записью проверить в браузере.

### Источники и формат ссылок (должны быть точными)

| Тип | Рекомендуемый URL-формат | Примечание |
|-----|-------------------------|-----------|
| Российский патент | `https://www.fips.ru/ru/patents/patent/{номер}` (ФИПС) или `https://patents.google.com/patent/RU…/ru` | Приоритет — ФИПС; Google Patents как дополнительный источник |
| Международный патент (PCT) | `https://patentscope.wipo.int/search/en/detail.jsf?docId=RU…` | WIPO PATENTSCOPE |
| Европейский патент | `https://register.epo.org/search?selection=REG&query=EP…` | EPO Register |
| Научная статья (КиберЛенинка) | `https://cyberleninka.ru/article/n/…` | КиберЛенинка |
| Научная статья (eLibrary) | `https://www.elibrary.ru/…` | eLibrary.ru |
| Научная статья (Scholar) | Scholar-ссылка или **`https://doi.org/10.xxxx/...`** | DOI приоритетнее |
| Preprint (arXiv) | `https://arxiv.org/abs/2008.09213` | arXiv |

В конце — **резюме поиска** и **существенное отличие** данного решения от предшествующего уровня техники, в согласии с 1.1 и разделом 1.2.

## Привычки ведения записей

Для удобства записи в заявку: фиксировать номер публикации, название, **переработанную** 1–2-строчную схему (при наличии **`abstract`** — согласованную с ним); **каждая запись — отдельная строка или таблица** с **«ссылка на источник»**; в 1.1 «Обзор поиска» — кратко указать, использовался ли **`fips_epub_search.py`**, было ли **снижение до Google Scholar**. Избегать копирования формул изобретения или полного копирования абстрактов.
