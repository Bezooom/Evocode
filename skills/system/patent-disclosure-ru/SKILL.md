# Поиск патентных решений и генерация технической заявки

Навык охватывает полный цикл: **поиск патентных решений** → **сравнительный анализ** → **генерация технической заявки** → **самопроверка**; пошаговые инструкции в **`prompts/`**, перед каждым шагом **`Read`** соответствующего файла. Соответствие шагов — в таблице «Соответствие файлов prompts».

## Окружение и соглашения

- **Язык**: по умолчанию — язык пользователя; патентные и юридические термины — по российским стандартам.
- **Финальные диаграммы (Шаг 7)**: **3.2** и **3.4** — fenced **mermaid**; запуск, установка **`mmdc`** и правила понижения — в строке «Финальная доставка заявки» таблицы «Инструменты и источники данных» и в **`tools/README.md`** (ASCII-диаграммы или PlantUML не требуются).

---

## Условия активации

Активируйте навык при любом из следующих условий:

- Явный запрос: поиск патентных решений, патентные решения, техническая заявка, заявка, патентная заявка, проверка новизны, сравнение с предшествующим уровнем техники и т.д.
- Косвенные запросы: например `/patent-disclosure-skill`, `/patent-disclosure`, `/заявка`
- **Итеративный режим (по распознаванию намерения)**: когда намерение пользователя явно связано с **продолжением работы над уже готовой технической заявкой или предыдущей итерацией** (изменение разделов, добавление примеров реализации, добавление материалов, исправление параметров/фактов, корректировка формулировок и т.д.), **не требуется** от пользователя использовать слово «итерация», и **не нужно** спрашивать — Agent должен **`Read`** **`prompts/iteration_context.md`**, затем **`Read`** `prompts/merger.md` (акцент на **добавление новых материалов, расширение**) или `prompts/correction_handler.md` (акцент на **исправление ошибок, несоответствие фактам и стилю**), **строго следовать «вратам выполнения»** в начале этих файлов (**выполнить первым, не пропускать**) **завершить слияние или исправление**, **сохранить в новый файл**: **`{название_изобретения}_{YYYYMMDDHHmmss}.md`** и同名 **`.docx`** (то же правило именования, что и при первичной финальной доставке, см. **`disclosure_builder.md` §7.3 пункт 5**), **не перезаписывать** старый черновик (если пользователь явно не просит). **Запрещено** при подтверждённом намерении итерации по умолчанию возвращаться к Шагу 3–4 полному анализу патентных решений (если пользователь явно не просит переискать). Если в диалоге **уже есть** путь к технической заявке, вложение или предыдущий черновик — приоритет итерации.

## Инструменты и источники данных

Используйте возможности по задаче; названия конкретных инструментов зависят от окружения Agent.

Если в зоне сканирования есть **Word (.docx)** или **PowerPoint (.pptx)**, перед Шагом 2 **обязательно** преобразуйте их в Markdown с помощью скриптов **`docx_to_md.py`** / **`pptx_to_md.py`** из этого репозитория; зависимости: **`pip install -r requirements.txt`**, команды и описания — в таблице ниже.

### Рекомендованные способы для типовых задач

| Задача | Рекомендованный способ |
|--------|----------------------|
| Загрузка пошаговых инструкций | **`Read`** → `${CLAUDE_SKILL_DIR}/prompts/*.md`, см. таблицу ниже |
| Чтение кода, проектных документов, PDF, изображений | Чтение файлов; для крупных репозиториев — сначала поиск/семантический поиск, затем детальное чтение |
| Word (.docx) → Markdown + извлечение изображений (до сканирования) | `Bash` → `python3 ${CLAUDE_SKILL_DIR}/tools/docx_to_md.py --input {path}.docx --output {dir}/{name}.md`; изображения по умолчанию в `{name}_media/` рядом с `.md`; требуется `pip install -r requirements.txt` (включая mammoth); сложный макет — экспортируйте PDF/MD внутри офиса |
| PowerPoint (.pptx) → Markdown + извлечение изображений (до сканирования) | `Bash` → `python3 ${CLAUDE_SKILL_DIR}/tools/pptx_to_md.py --input {path}.pptx --output {dir}/{name}.md`; по умолчанию `{name}_media/`; требуется `pip install -r requirements.txt` (включая python-pptx); **старые `.ppt` не поддерживаются**, сначала сохраните как `.pptx`; диаграммы/SmartArt могут быть доступны только из заметок или дополнительного экспорта |
| Список каталогов, поиск файлов по имени | Перечень каталога / поиск по имени файла |
| Проверка новизны в сети (Шаг 5) | Перед запуском **`Read`** `prompts/prior_art_search.md`. **ФИПС (Роспатент)**: приоритетно **`Bash`** запустить `fips_epub_search.py`; **обязательно до генерации команды** собрать **2–8 релевантных семантических блоков**; **запускать последовательно**, **каждый блок — отдельный вызов**, **самостоятельно объединить** результаты `EPUB_HITS_JSON` (не передавать несколько аргументов за один вызов, см. этот prompt). Одношаговая загрузка + разбор, **без записи HTML на диск**; требуется **`pip install -r tools/requirements-fips.txt`** и **`python -m playwright install chromium`**. **`abstract`** — обязательное поле, использовать строго по этому prompt. Для полных предложений AND или сохранения HTML используйте `fips_epub_crawler.py`; при ошибках или отсутствии результата — **WebSearch** |
| Финальная доставка заявки (**обязательно** .md + .docx) | **3.2** и **3.4** — только fenced ``mermaid``, **не** ASCII-диаграммы. Для финальной доставки запускайте **`tools/mermaid_render.py`**: mermaid → PNG (при ошибке блок сохраняется в围栏), затем по умолчанию генерируется **.docx**; при ошибке Word — по stderr запустите **`md_to_docx.py`**. См. **`tools/README.md`** |
| Сохранение пути к заявке | Записать в путь, указанный пользователем; если не указан — предложить `./outputs/{идентификатор_проекта}/`; **любая доставленная** `.md` / `.docx` должна быть с именем **`{название_изобретения}_{YYYYMMDDHHmmss}`** (§7.3 пункт 5, **включая первичную финальную и итерации**), не перезаписывать старые файлы по умолчанию; `outputs/` — по умолчанию в `.gitignore` |

## Основной процесс (порядок выполнения)

1. **`Read`** `intake.md` → Шаг 1  
2. **`Read`** `project_scan.md` → Шаг 2  
3. **`Read`** `patent_points_analyzer.md` → Шаги 3–4  
4. **`Read`** `prior_art_search.md` → Шаг 5  
5. **`Read`** `disclosure_preview.md` → Шаг 6; пользователь может пропустить  
6. **`Read`** `disclosure_builder.md` и **`Read`** `template_reference.md` → Шаг 7 (**первичная доставка** `.md`/`.docx` также с именем **`{название_изобретения}_{YYYYMMDDHHmmss}`**, §7.3 пункт 5)  
7. **`Read`** `disclosure_self_check.md` → внутреннее выполнение Шага 8, исправление и доставка  

**Запрещено**: включать чеклист самопроверки в тело заявки; самопроверка только внутренняя.

## Итеративный режим (кратко)

**Активация**: по **естественному языковому намерению** пользователя (см. «Условия активации» выше), **не требуется** ключевое слово, **по умолчанию не** прерывать пользователя вопросом о итерации.

- **Добавление материалов / расширение разделов**: `Read` → `iteration_context.md` → `merger.md`; результат — **сохранить в новый** `.md`/`.docx` с timestamp (§7.3 пункт 5); **добавить** `Лог_итераций_заявки.md` (скрипт `iteration_dialog_log.py` или вручную); после завершения **обязательно** вывести «Сводка слияния» для архива  
- **Указание на ошибку / несоответствие фактам или параметрам**: `Read` → `iteration_context.md` → `correction_handler.md`; результат — **сохранить в новый** `.md`/`.docx` с timestamp; **добавить** запись в лог; после завершения **обязательно** вывести «Сводка исправлений» для архива  

Самопроверка `disclosure_self_check.md` из Шагов 7→8 всё равно выполняется внутренне для новой версии.
