# Платформа обработки данных — архитектура планирования пакетных задач (вымышленный пример)

> **Заявление**: Этот документ предназначен только для примера патентного навыка этого репозитория, не соответствует никакому реальному продукту.

## 1. Общая структура

- **Планировщик (Scheduler)**: поддерживает глобальную очередь задач, сортирует по приоритету; периодически отправляет задачи на рабочие узлы.
- **Рабочий узел (Worker)**: выполняет конкретные задачи, отчитывает CPU/память/IO и другие метрики.
- **Центр конфигурации**: приоритеты задач, пороговые значения ресурсов, теги аффинности и другие статические правила.

## 2. Текущие проблемы (со стороны бизнеса)

- Гетерогенные узлы кластера: некоторые узлы имеют высокую вычислительную мощность и большой локальный SSD, другие — высокую IO-полосную пропускную способность, но обычный CPU.
- Проблема: **задачи с большим объёмом** долго занимают высокопроизводительные узлы, **малые задачи** слишком долго ожидают на узлах с низкой мощностью; только статический приоритет не отражает **реальную** доступность ресурсов.
- Конфигурация уже имеет приоритет и простые пороговые значения нагрузки, но **не** объединяет «характеристики требований задачи» с **многомерным профилем ресурсов узла** для вторичной сортировки.

## 3. Цели (согласованы с intake)

- Снизить вероятность «голодания» задач с длительным временем ожидания.
- Допустимая **минутная** задержка планирования; избегание секундной частой перепланировки, приводящей к колебаниям.

## 4. Разделение модулей (согласовано с диаграммой заявки)

| Модуль | Ответственность |
|--------|-----------------|
| Агрегация профиля ресурсов | Сбор метрик узла, скользящее окно сглаживания |
| Оценивание соответствия | Вектор требований задачи × профиль узла → оценка соответствия |
| Перепланировка с лимитом | При соблюдении интервала и порога изменения оценки — перепланировка подмножества окна очереди |
| Отправка | По порядку, повтор при ошибке (опущено) |
| Heartbeat-контракт | Лёгкий heartbeat узла, для обнаружения зависания и триггера миграции (вторичный признак) |

## 5. Ограничения текущего решения

- Сортировка по статическому приоритету не учитывает **реальную** доступность ресурсов узла в момент планирования.
- Нет механизма **многомерного профиля** узла (CPU, IO, локальный SSD) в оценке соответствия.
- Частые перепланировки без ограничения частоты приводят к **колебаниям** и дублированию отправки.
