# АЗТП-СКИЛЛ — Техническое задание на АС и ПО (ГОСТ 34.201-2020 + ГОСТ 19.101-2024 + ГОСТ Р 56939-2024)

Агентская система разработки ТЗ для автоматизированных систем и программного обеспечения
по ГОСТ 34.201-2020 (вместо ГОСТ 34.201-89), ГОСТ 19.101-2024 (вместо ГОСТ 19.101-79) и ГОСТ Р 56939-2024 (вместо ГОСТ Р 56939-2016).

**Примечание:** ГОСТ 34.201-2020 принят Межгосударственным советом по стандартизации
(протокол от 22 декабря 2020 г. № 58), введён в действие с 1 января 2022 г.
ГОСТ Р 56939-2024 введён для обеспечения безопасной разработки (SDL) и включает требования по статическому, динамическому анализу, фаззингу и моделированию угроз.

---

## Когда использовать

- Создание ТЗ на автоматизированную систему
- Разработка ТЗ на программное обеспечение
- Подготовка ТЗ для государственного заказчика (военное, гражданское)
- Разработка ТЗ для интеграции с существующими системами
- Подготовка ТЗ для программных продуктов
- Нормоконтроль ТЗ по ГОСТ 34 и ГОСТ 19

### 1.1 Сбор исходных данных у заказчика

## Фаза 2: Контекст и требования

### 2.1 Анализ требований

1. **Бизнес-требования** — цели, обоснование
2. **Функциональные требования** — что должна делать система
3. **Нефункциональные требования** — производительность, надёжность, безопасность
4. **Требования к данным** — объём, структура, хранилище
5. **Требования к интерфейсу** — пользовательский и аппаратный
6. **Требования к интеграции** — API, протоколы, форматы данных
7. **Ограничения** — бюджет, сроки, технологии
8. **Предположения и зависимости**

**Примеры:**

> **FR-001.** Система должна обеспечивать автоматическую обработку не менее 1000 документов в час.
>
> **NFR-001.** Время отклика системы ≤ 3 секунд для 95% запросов.
>
> **NFR-002.** Доступность системы (uptime) ≥ 99,5%.

|------|----------------|------------|

| Эксплуатант | Поддержание | Удобство, документация |
| Пользователь | Использование | Интуитивность, скорость |

### 3.4 Требования к функциональным характеристикам

| Требование | Описание | Метод верификации |
|------------|----------|-------------------|
| Производительность | Обработка X документов/час | Тестирование |
| Пропускная способность | ≥ X пользователей | Нагрузочный тест |
| Точность | Погрешность ≤ Y% | Верификация |
| Надёжность | MTBF ≥ Z часов | Мониторинг |
| Безопасность | Уровень защиты W | Аудит безопасности |

**Примеры:**

> **ФТ-1.** Система обработки документов должна обеспечивать:
> - обработку ≥ 1000 документов в час;
> - одновременную работу ≥ 50 пользователей;
> - время отклика ≤ 3 секунд (95% запросов);
> - время восстановления ≤ 1 часа.

### 3.5 Требования к надёжности

| Параметр | Значение | Метод определения |
|----------|----------|-------------------|
| MTBF | ≥ 720 часов | Мониторинг |
| MTTR | ≤ 1 час | Тестирование |
| Доступность | ≥ 99,5% | Мониторинг |
| Сбоев/год | ≤ 10 | Мониторинг |

### 3.6 Требования к информационной безопасности (в т.ч. ГОСТ Р 56939-2024)

| Требование | Описание |
|------------|----------|
| Аутентификация | Многоуровневая (пароль + 2FA) |
| Авторизация | Ролевой доступ (RBAC) |
| Шифрование | TLS 1.3, AES-256 для данных |
| Журналирование | Loglevel INFO, хранение ≥ 1 года |
| Резервирование | Бэкап ежедневно, RPO ≤ 1 час |
| **SDL (ГОСТ Р 56939)** | Внедрение процессов безопасной разработки, включая управление уязвимостями |
| **Анализ кода** | Интеграция SAST и DAST в пайплайн CI/CD |
| **Фаззинг** | Динамическое тестирование нетипичными/некорректными данными |
| **Моделирование угроз** | Регулярный анализ архитектуры на предмет векторов атак (STRIDE и др.) |

## Антипаттерны ТЗ

1. **Отсутствие оснований для разработки** — нет ссылки на приказ, договор, договорённость
2. **Размытые требования** — «должна быть быстрая обработка» вместо «≤ 3 сек»
3. **Нет требований к надёжности** — MTBF, MTTR, доступность не указаны
4. **Нет требований к безопасности** — аутентификация, авторизация, шифрование
5. **Нет критериев приёмки** — непонятно, что считать готовым продуктом
6. **Нет требований к интеграции** — не описана связь с существующими системами
7. **Игнорирование безопасной разработки (SDL)** — отсутствие требований к анализу уязвимостей, SAST/DAST и фаззингу по ГОСТ Р 56939-2024
8. **Неполный перечень документов** — отсутствуют обязательные документы по ГОСТ
9. **Нет требований к программным документам** — не указан состав ПИ
10. **Нет требований к аппаратному обеспечению** — не указаны требования к серверам
11. **Нет требований к документации** — не указан состав документации
12. **Нет гарантийных обязательств** — не определены сроки и условия
13. **Нет приложения** — отсутствуют справочные данные и требования
14. **Неверные обозначения** — коды не по ГОСТ 34.003-90
15. **Нет стадий работ** — не определён порядок работ
16. **Нет требований к тестированию** — нет плана испытаний

## Pipeline разработки ТЗ

```
Сбор требований → Контекст → ТЗ (ГОСТ 34 / 19) → Документация → Испытания → Нормоконтроль → Утверждение
```

### Работа с контекстом
- При разработке каждого раздела — загружать соответствующий reference файл
- Не загружать все reference одновременно
- Использовать meta.yaml для согласованности данных
- После каждого этапа — запускать валидатор
- Перед утверждением — проверять чек-лист нормоконтроля
