## 🛠️ Детальные шаги интеграции

### 1. Интеграция Smart Router v2 (из `Router/`)
Текущий роутер Эвокод использует простой набор регулярных выражений и ключевых слов для определения сложности задачи (`simple`/`complex`). Проект `Router/` содержит профессиональный классификатор.

*   **Что читать:** `Router/router/classifier.py` и `Router/router/cost_optimizer.py`.
*   **Что извлечь:** Алгоритм классификации текстового запроса (анализ токенов, длина запроса, когнитивная нагрузка).
*   **Как адаптировать (Python → TypeScript):**
    *   Создать `src/router/ml-router.ts`.
    *   Реализовать легковесный TF-IDF векторайзер на TypeScript (без внешних тяжелых зависимостей) или использовать модуль `natural` для классификации намерений (intent classification).
    *   Реализовать логику `fallback-цепочек` (если локальный инференс падает с ошибкой, запрос автоматически маскируется DLP и отправляется в OpenRouter).
*   **Как протестировать:** Обновить [smart-router.test.ts](file://$EVOCODE_ROOT  # path to clone/tests/unit/smart-router.test.ts), подавая запросы разной сложности (например, "исправь опечатку" vs "напиши архитектуру распределённого кэша") и проверяя корректность выбора модели.

### 2. Подключение библиотеки навыков (из `Skills/`)
Проект `Skills/` содержит 247 готовых файлов `SKILL.md` во множестве категорий (coding, DevOps, Data Science, QA и др.).

*   **Что читать:** `Skills/skills/` (все поддиректории).
*   **Что извлечь:** Markdown-файлы навыков с YAML-frontmatter.
*   **Как адаптировать:**
    *   Поскольку Skill Sync Engine в Эвокод уже умеет работать с GitHub API, мы можем:
        1. Либо скопировать файлы напрямую в `skills/system/` для оффлайн-работы.
        2. Либо настроить конфиг Эвокод (`src/core/config.ts`) на локальный или удалённый репозиторий `skills` как источник синхронизации.
*   **Как протестировать:** Запустить `POST /chat` с запросом, требующим специфического навыка (например, "напиши конфиг для nginx с кэшированием"), и проверить, обогащается ли контекст нужным навыком из директории `skills/system/`.

### 3. Интеграция OpenAI-совместимого эндпоинта (из `openai_proxy.py`)
Файл `openai_proxy.py` реализует легковесный Flask-прокси, транслирующий вызовы в формате OpenAI API в формат, понятный локальному `llama-server`.

*   **Что читать:** `openai_proxy.py` в корне Projects.
*   **Что извлечь:** Правила трансляции JSON-структур (из `v1/chat/completions` в `/completion` llama.cpp).
*   **Как адаптировать:**
    *   Добавить новые роуты в `src/index.ts` (или вынести в `src/server/routes.ts`):
        *   `POST /v1/chat/completions`
        *   `GET /v1/models`
    *   Реализовать маппинг параметров: `messages` (OpenAI) транслируются в текстовый промпт с соответствующим шаблоном (ChatML, LLaMA-3 и др.) для llama.cpp.
*   **Как протестировать:** Выполнить curl-запрос в формате OpenAI к серверу Эвокод на порт 8081 и убедиться, что он возвращает валидный OpenAI-ответ.

### 4. Внедрение WebSocket Streaming (из `Messenger/`)
Для плавной генерации кода в UI необходим посимвольный/поток токенов (streaming).

*   **Что читать:** `Messenger/src/ws/` или `Messenger/src/server.ts`.
*   **Что извлечь:** Конфигурацию WebSocket-сервера (`ws` или `socket.io`) и обработку событий отправки чанков текста.
*   **Как адаптировать:**
    *   Инсталлировать библиотеку `ws` в Эвокод (`npm install ws @types/ws`).
    *   Создать `src/server/websocket.ts` для параллельного запуска WebSocket-сервера на порту 8082 (или на том же 8081 с апгрейдом протокола).
    *   В `InferenceEngine` настроить передачу callback-функции `onToken` в метод инференса, которая будет слать чанки через сокет.
*   **Как протестировать:** Создать простой html-клиент (`demo.html`) и проверить плавное отображение ответа.

### 5. Дообучение мини-нейросети (из `Neurocontrol/`)
Это наиболее сложный, но самый ценный этап (реализация пользовательской адаптации).

*   **Что читать:** `Neurocontrol/training/lora_adapter.py` и `Neurocontrol/api/training_api.py`.
*   **Что извлечь:** Пайплайн запуска PyTorch/HuggingFace тренировки с использованием LoRA (Low-Rank Adaptation) и экспорта готовых весов в формате GGUF.
*   **Как адаптировать:**
    *   Поскольку Эвокод написан на Node.js, запускать PyTorch напрямую из JS неэффективно.
    *   Необходимо реализовать запуск Python-скрипта дообучения в качестве фонового процесса через `child_process.spawn`.
    *   В Эвокод создается модуль `src/training/fine-tune.ts`, который подготавливает датасет (лог локальных исправлений кода пользователя), генерирует JSONL-файл и запускает скрипт обучения.
    *   После обучения веса LoRA-адаптера подключаются к `llama-server` при запуске (`--lora lora_weights.bin`).
*   **Как протестировать:** Проверить генерацию обучающего датасета из логов взаимодействия и убедиться, что процесс обучения стартует и успешно компилирует GGUF/LoRA адаптер.

## ⚠️ Риски и митигация

1.  **Различие языков программирования (Python ↔ TypeScript):**
    *   *Риск:* Большинство ML-компонентов (`Neurocontrol`, `Router`, `Agent`) написаны на Python.
    *   *Решение:* Переносить на TypeScript только легкую логику (роутинг, парсинг, коннекторы). Тяжелые процессы (обучение моделей, скрапинг Selenium) запускать как изолированные дочерние процессы (spawn) или микросервисы.
2.  **Потребление памяти (VRAM/RAM):**
    *   *Риск:* Одновременный запуск `llama-server`, RAG-индексации и фонового LoRA-дообучения может перегрузить GPU/RAM.
    *   *Решение:* Использовать планировщик задач. Запускать обучение LoRA только в простое системы (например, ночью) и временно останавливать `llama-server` на время обучения для освобождения VRAM.
3.  **Безопасность данных (DLP):**
    *   *Риск:* При вызове внешних MCP-инструментов или скрапинге через `Aist` конфиденциальные данные могут утечь во внешнюю сеть.
    *   *Решение:* Все вызовы инструментов и MCP-серверов должны проходить через локальный прокси-слой DLP Guard. Любые исходящие сетевые запросы блокируются, если обнаружены совпадения с правилами маскирования.
