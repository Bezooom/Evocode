# Сценарии использования (User Stories) для «Эвокод»

## 1. Базовые сценарии

### US-01: Первый запуск и настройка
**Как** новый пользователь, **я хочу** чтобы Эвокод запустился и настроился без вмешательства, **чтобы** начать работать сразу.

**Критерии приёмки:**
- Приложение запускается, показывает приветствие
- Предлагает скачать рекомендованную GGUF-модель (LLaMA-3.1-8B-Q4_K_M)
- После загрузки модели показывает пример чата
- Настраивает русскую локализацию по умолчанию
- Запускает Skill Sync Engine и показывает результат

### US-02: Локальный чат без интернета
**Как** разработчик, **я хочу** чтобы Эвокод работал без интернета, **чтобы** не зависеть от облачных сервисов.

**Критерии приёмки:**
- Отключаю интернет
- Отправляю запрос в чат
- Получаю ответ от локальной модели без задержек
- В строке состояния вижу бейдж «Локально»
- При подключении интернета Skill Sync Engine обновляет навыки

### US-03: Автодополнение кода
**Как** разработчик, **я хочу** чтобы Эвокод предлагал inline-подсказки, **чтобы** ускорить написание кода.

**Критерии приёмки:**
- Ввожу код в редакторе
- Через < 100ms появляется предложение
- Предложение корректно завершает строку/блок
- При игнорировании предложение исчезает
- При принятии предложение вставляется в код

## 2. Сценарии роутинга

### US-04: Сложная задача уходит в облако
**Как** разработчик, **я хочу** чтобы сложные задачи автоматически отправлялись в облако, **чтобы** получить лучшее качество.

**Критерии приёмки:**
- Отправляю запрос с большой сложностью (complexity_score > 0.7)
- Вижу в чате бейдж «Облако»
- Получаю ответ от OpenRouter
- В логах вижу, что запрос прошёл через DLP Guard
- Секреты в промпте маскируются

### US-05: Fallback при сбое локальной модели
**Как** разработчик, **я хочу** чтобы при сбое локальной модели Эвокод переключился в облако, **чтобы** не потерять прогресс.

**Критерии приёмки:**
- Локальная модель падает (нехватка VRAM)
- Эвокод автоматически переключается в облако
- Получаю уведомление о fallback
- После восстановления модели запросы снова идут локально
- В логах вижу запись о fallback

### US-06: Ручное переключение на облако
**Как** разработчик, **я хочу** чтобы я мог вручную переключить запрос в облако, **чтобы** использовать конкретную модель.

**Критерии приёмки:**
- Нажимаю кнопку «В облако» в чате
- Запрос отправляется в OpenRouter с выбранной моделью
- Получаю ответ от облачной модели
- В строке состояния вижу текущий провайдер

## 3. Сценарии синхронизации

### US-07: Ежедневное обновление навыков
**Как** разработчик, **я хочу** чтобы Эвокод ежедневно обновлял навыки из GitHub, **чтобы** система была актуальной.

**Критерии приёмки:**
- При запуске или по расписанию (03:00) Skill Sync Engine опрашивает GitHub
- Находит новые и изменённые навыки
- Применяет обновления в системный каталог
- Сохраняет предыдущую версию в бэкап
- Показывает лог обновлений в UI
- Если нет интернета — пропускает и записывает «offline, skip»

### US-08: Пользовательские оверрайды не перезаписываются
**Как** разработчик, **я хочу** чтобы мои изменения в навыках не перезаписывались при автообновлениях, **чтобы** сохранить свою кастомизацию.

**Критерии приёмки:**
- Создаю оверрайд в `~/.config/evocode/skills/user/`
- Автообновление находит изменения в системном навыке
- Системный навык обновляется, оверрайд остаётся
- При загрузке оверрайд перекрывает системный навык
- В логах вижу, что оверрайд применён

### US-09: Откат обновлений
**Как** разработчик, **я хочу** чтобы я мог откатить обновление навыка, **чтобы** вернуться к предыдущей версии.

**Критерии приёмки:**
- Обновляю навык, но новая версия не работает
- Нажимаю кнопку «Откатить» в UI
- Навык возвращается к предыдущей версии из бэкапа
- В логах вижу запись об откате
- Навык работает корректно

## 4. Сценарии безопасности

### US-10: DLP Guard маскирует секреты
**Как** разработчик, **я хочу** чтобы Эвокод маскировал API-ключи и пароли перед отправкой в облако, **чтобы** не утекли секреты.

**Критерии приёмки:**
- Отправляю промпт с API-ключом (например, `sk-abc123`)
- DLP Guard находит ключ и маскирует его в `[REDACTED_API_KEY]`
- Запрос отправляется в OpenRouter без секрета
- В логах вижу маскированное значение
- В ответе получаю корректный результат

### US-11: Блокировка критических утечек
**Как** разработчик, **я хочу** чтобы Эвокод блокировал отправку промптов с критическими утечками, **чтобы** предотвратить потерю данных.

**Критерии приёмки:**
- Отправляю промпт с полным дамп-конфигом (пароли, ключи)
- DLP Guard находит критическую утечку
- Отправка блокируется, появляется уведомление
- Я могу разрешить отправку или скорректировать промпт
- После разрешения запрос уходит в облако

### US-12: Локальный вывод не покидает устройство
**Как** разработчик, **я хочу** чтобы данные локального вывода оставались на устройстве, **чтобы** обеспечить приватность.

**Критерии приёмки:**
- Отключаю интернет
- Отправляю промпт в чат
- Получаю ответ без отправки данных в облако
- В логах нет записей о сетевых запросах
- В настройках вижу, что телеметрия отключена

## 5. Сценарии интеграции

### US-13: Запуск субагентов
**Как** разработчик, **я хочу** чтобы Эвокод мог запускать субагентов, **чтобы** делегировать сложные задачи.

**Критерии приёмки:**
- Отправляю запрос, который требует субагента
- Эвокод определяет тип задачи и загружает навык
- Запускает субагента через `task` инструмент
- Субагент выполняет работу и возвращает результат
- Оркестратор агрегирует результат и сообщает пользователю

### US-14: Работа с MCP-серверами
**Как** разработчик, **я хочу** чтобы Эвокод мог вызывать MCP-серверы, **чтобы** использовать внешние инструменты.

**Критерии приёмки:**
- Добавляю MCP-сервер через UI
- Отправляю запрос, который вызывает MCP-инструмент
- Эвокод обращается к MCP Host
- MCP Host выполняет инструмент (чтение файлов, терминал)
- Получаю результат в чате

### US-15: Local RAG обогащает контекст
**Как** разработчик, **я хочу** чтобы Эвокод искал релевантный код в проекте, **чтобы** улучшить контекст для LLM.

**Критерии приёмки:**
- Открываю проект в Эвокод
- Local RAG фоновое индексирует файлы
- Отправляю запрос о конкретном классе
- Эвокод извлекает релевантные куски из векторной БД
- Контекст подмешивается в промпт без отправки данных в облако

## 6. Сценарии обновления

### US-16: Обработка breaking changes
**Как** разработчик, **я хочу** чтобы Эвокод предупреждал о breaking changes, **чтобы** я мог подготовиться.

**Критерии приёмки:**
- Синхронизирую навык с `breaking: true`
- Получаю уведомление о breaking changes
- Вижу описание изменений и рекомендации по миграции
- Могу применить обновление или отложить
- После применения навык работает корректно

### US-17: Каналы обновлений (stable/beta/canary)
**Как** разработчик, **я хочу** чтобы Эвокод предлагал разные каналы обновлений, **чтобы** я мог выбрать между стабильностью и новизной.

**Критерии приёмки:**
- В настройках выбираю канал обновлений
- Для stable — получаю проверенные обновления
- Для beta — получаю новые фичи с возможными regressions
- Для canary — получаю самые свежие обновления
- Переключение каналов работает без потери данных

## 7. Сценарии тестирования

### US-18: Автоматические тесты при обновлении
**Как** разработчик, **я хочу** чтобы Эвокод запускал тесты при обновлении навыков, **чтобы** убедиться в корректности.

**Критерии приёмки:**
- Обновляю навык
- Эвокод запускает unit-тесты для навыка
- Если тесты проходят — навык применяется
- Если тесты падают — навык откатывается
- В логах вижу результат тестирования

### US-19: Ночные сборки (nightly build)
**Как** разработчик, **я хочу** чтобы Эвокод мог получать ночные сборки, **чтобы** тестировать новые фичи.

**Критерии приёмки:**
- Включаю ночные сборки в настройках
- Каждый день получаю обновление до последней nightly
- Вижу в логах информацию о сборке
- Могу откатиться на предыдущую сборку

## 8. Сценарии деплоя

### US-20: Развёртывание на Astra Linux
**Как** системный администратор, **я хочу** чтобы Эвокод работал на Astra Linux, **чтобы** использовать его в защищённом контуре.

**Критерии приёмки:**
- Устанавливаю Эвокод на Astra Linux
- Приложение запускается без ошибок
- Локальная модель работает корректно
- Skill Sync Engine синхронизируется с GitHub
- В логах нет ошибок совместимости

### US-21: Развёртывание в корпоративном контуре
**Как** системный администратор, **я хочу** чтобы Эвокод мог работать через корпоративный прокси, **чтобы** использовать его в защищённой сети.

**Критерии приёмки:**
- Настраиваю HTTP/SOCKS5 прокси в настройках
- Эвокод подключается к OpenRouter через прокси
- Skill Sync Engine синхронизируется через прокси
- API-ключи хранятся в корпоративном Keychain
- В логах нет ошибок подключения
